git branches that are protected: alpha/*, beta/*, master, release/* the alpha, beta and release branches share the same version number as the name so alpha/1.0.1.x beta/1.0.1.x release/1.0.1.x

new alpha branch is spun off from master it has the current version of 3 plus a 0 for x , so 1.0.1.0 issues solved in alpha are done in topic branches, then put in master, and then new alpha branch is made with x bumped up so alpha/1.0.1.1. Old alpha branch of 1.0.1.0 is deleted off origin

once the current alpha branch looks stable enough a new beta branch is spun off that alpha, and has the same version of 4, so beta/1.0.1.1 when during beta testing issues are fixed in topics, they are pushed to master , and a new alpha branch is made with x (2 for example) bumped up again (and the older alpha removed) the new beta branch from the fixes in alpha branch is created (so now beta/1.0.1.2 is here and the older beta/1.0.1.1 is removed from origin) keep cycling like that until we get a beta that passes ok (so for example beta/1.0.1.23) when release is ready, the release branch is spun off from the beta branch , made release/1.0.1.23 and its distributed and the master branch has that tag now at the last commit The release branchâ€™s lifetime ends when its replaced by another release.

pre-alpha branches escape all of these rules, as development is too fast and chaotic for all this nonsense of well established work flow. Chaos rules!
